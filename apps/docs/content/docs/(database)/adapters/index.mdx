# Adapters

O sistema de adapters separa a API de documentos do backend de persistência.

Em outras palavras, `Database`, `Schema` e `Collection` continuam os mesmos, enquanto o storage pode ser local, MongoDB, Mongoose ou qualquer implementação compatível com `StorageAdapter`.

## Comparação rápida

| Adapter | Quando usar | Dependências extras | Persistência de schema | Lifecycle próprio |
| --- | --- | --- | --- | --- |
| Default (BSON local) | desenvolvimento local, CLIs, apps pequenas e médias | nenhuma | `_schema.json` em disco | flush local |
| MongoDB | quando você quer usar o driver oficial | `mongodb` | `_tile_schemas` opcional | `manageClient` |
| Mongoose | quando o projeto já depende de Mongoose | `mongoose` | `_tile_schemas` opcional | `manageConnection` |
| Custom | integrações próprias | depende do seu backend | depende do adapter | depende do adapter |

## Formas de importar

<Tabs items={["subpaths", "barrel"]}>
  <Tab value="subpaths">

```ts
import { defaultAdapter } from "@tile.js/database";
import { mongodbAdapter } from "@tile.js/database/adapters/mongodb";
import { mongooseAdapter } from "@tile.js/database/adapters/mongoose";
```

  </Tab>

  <Tab value="barrel">

```ts
import {
  defaultAdapter,
  mongodbAdapter,
  mongooseAdapter,
} from "@tile.js/database/adapters";
```

  </Tab>
</Tabs>

<Callout type="warning" title="Semântica do Tile continua valendo">
Os adapters trocam o backend, mas não transformam a biblioteca em um ODM nativo do MongoDB ou do Mongoose. Filtros continuam simples, e operações como `find()`, `countDocuments()`, `deleteMany()` e `paginate()` seguem a semântica do Tile.
</Callout>

## O que permanece igual entre adapters

- `Schema` continua sendo o centro da validação e normalização
- `_id`, `__v` e timestamps continuam seguindo as regras do Tile
- `create`, `findOne`, `updateOne`, `upsert`, `deleteMany` e `paginate` continuam com a mesma API
- `collection()` e `model()` continuam funcionando da mesma forma
