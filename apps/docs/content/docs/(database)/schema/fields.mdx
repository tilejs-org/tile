# Campos e validação

## Tipos suportados

O Tile entende estes tipos internos:

- `String`
- `Number`
- `Boolean`
- `Date`
- `Array`
- `Object`

Você pode declarar usando construtores:

```ts
const schema = new Schema({
  name: String,
  age: Number,
  active: Boolean,
  createdAtSource: Date,
  tags: Array,
  profile: Object,
});
```

## Normalização e validação

```ts
const schema = new Schema({
  email: {
    type: String,
    required: true,
    lowercase: true,
    trim: true,
  },
  visits: {
    type: Number,
    min: 0,
  },
  role: {
    type: String,
    enum: ["admin", "member"],
  },
});
```

Nesse exemplo, o Tile:

- normaliza `email` com `lowercase` e `trim`
- rejeita `visits < 0`
- rejeita valores fora do `enum`

## Objetos aninhados viram dot-paths

```ts
const schema = new Schema({
  profile: {
    name: String,
    email: { type: String, unique: true },
  },
});
```

Internamente, o schema passa a enxergar:

- `profile.name`
- `profile.email`

Isso vale para:

- validação
- defaults
- updates
- filtros simples

Exemplo de filtro:

```ts
await database.member.findOne({
  "profile.email": "ada@example.com",
});
```

## Datas, arrays e objetos

- `Date` aceita `Date`, `string` ou `number`, desde que o valor resulte em uma data válida
- `Array` valida apenas se o valor é um array
- `Object` exige um objeto simples e rejeita arrays e `Date`

<Callout type="warning" title="Schema não é strict">
Campos fora do schema não são removidos automaticamente. Se você passar propriedades extras no documento ou em um update, elas podem ser persistidas normalmente.
</Callout>

<Callout type="warning" title="Limitações atuais importantes">
`Array` não descreve o tipo dos itens, e `options._id = false` hoje afeta a descrição do schema, mas os documentos continuam recebendo `_id` internamente.
</Callout>
